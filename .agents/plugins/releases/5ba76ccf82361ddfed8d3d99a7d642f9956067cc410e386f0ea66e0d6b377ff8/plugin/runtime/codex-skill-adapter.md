# Codex harness binding

Apply these bindings before following any ported Muster workflow. They override legacy harness names in the source instructions while preserving the workflow's domain logic, gates, approvals, and return contracts.

- Treat `Agent` and `Task` calls as Codex subagent dispatch. For a resolved `kind: agent` provider, call `collaboration.spawn_agent` with the ordinary `task_name`, `message`, and a bounded `fork_turns` value (`"none"` or a positive turn count, never `"all"`) plus `agent_type: "<exact chosen.id>"`. Codex rejects a named `agent_type` combined with a full-history fork because full-history agents inherit the parent's type/model/effort. `agent_type` is a Codex runtime extension and may be absent from a simplified displayed tool signature; include it anyway. The installed custom-agent TOML pins the role's model, reasoning effort, sandbox, and instructions. Only an actual rejected `spawn_agent` tool call proves the type unavailable; inspecting a displayed signature or assuming an omitted field is unsupported is not an attempt. If the call rejects that type, fail closed with a registration diagnostic rather than silently losing model policy through a general agent. For a resolved `kind: skill` provider, read `${PLUGIN_ROOT}/internal-skills/${chosen.id}/SKILL.md` and inject the workflow into a general subagent's brief; internal provider workflows are packaged there deliberately so Codex does not auto-register them. Use a general subagent directly for MCP/inline providers. Preserve dependency waves, parallel dispatch, isolation, and return contracts. Fall back to sequential work only when the source workflow explicitly permits it.
- Codex subagent dispatch has no cwd field. Every brief for worktree-scoped work must therefore name the absolute `WORKTREE CWD`, the absolute manifest/STATE paths inside that worktree, and require the child to use that directory as the `workdir` for its first verification command and every later tool call. The child must not read or mutate the parent checkout's `.muster` artifacts. Treat a parent-checkout/worktree manifest mismatch as a dispatch defect, not as authority to choose one silently.
- Treat `AskUserQuestion` as Codex's structured user-input mechanism when available. Otherwise ask one concise plain-text question and wait; never infer approval.
- Treat `TodoWrite` as Codex's plan or task interface.
- Treat `Skill` calls as explicit `$skill-name` invocations.
- Map `/muster:run` to `$muster-plan`, `/muster:autopilot` to `$muster-go`, `/muster:plan-backlog` to `$muster-plan-backlog`, `/muster:sprint` to `$muster-go-backlog`, and the diagnose, audit, runner, and capture commands to their matching `$muster-*` skills.
- `$ARGUMENTS` means the user's remaining prompt. `${PLUGIN_ROOT}` and `${CLAUDE_PLUGIN_ROOT}` both mean the installed Codex plugin root.
- Run every legacy `npx -y @adnova-group/muster ...` instruction through the self-contained bundled CLI instead: `node ${PLUGIN_ROOT}/runtime/muster.mjs ...`. Never contact the npm registry for a plugin workflow. Add `--codex` to `capabilities`, `match`, and `manifest validate` so provider and skill ids are resolved against the enabled Codex inventory.
- When a legacy workflow runs `muster capabilities`, use the bundled CLI with `capabilities --codex` so routing is based on enabled Codex plugins, skills, MCP servers, and profiles rather than Claude state.
- Translate `Read`, `Write`, `Edit`, `Bash`, `Grep`, and `Glob` to the available Codex tools while respecting the generated agent profile's read/write policy.
- When Claude Code itself is the actual subject of a workflow, retain that scope rather than claiming a nonexistent Codex equivalent.
- `muster install codex` merges Muster's lifecycle hooks into the selected project or user `hooks.json` layer. After the user trusts their exact definitions in `/hooks`, they provide context, diagnostics, and supported policy warnings. Codex 0.144 does not execute plugin-bundled hooks, so the plugin manifest must not be treated as hook activation evidence. Treat todo and spawn policy as advisory; every write-capable wave must still run in an isolated git worktree.

## Agent watch invariant

<!-- prompt-lint-disable GUARD-IDK-001: Explicit terminal conditions prevent abandoned live agents while preserving approval, HUMAN-HOLD, blocker, and merge-decision stops. -->

After every dispatch, retain every canonical agent id returned by `collaboration.spawn_agent` and immediately call `collaboration.wait_agent` with a timeout of at most 60 seconds. A message or completion receipt wakes the watch immediately. After each wake, process mailbox receipts first, call `collaboration.list_agents` exactly once to reconcile live state, dispatch any newly ready work whose dependencies are satisfied, and, while any agent remains live, immediately call `collaboration.wait_agent` again. A timeout is only a heartbeat: reconcile once and return to waiting; it is not completion. Never tight-poll `collaboration.list_agents` and never prompt the user merely because workers are still running.

Do not send the final answer, clear active run/wave state, or stop watching while live agents or executable steps remain. Stop only when all work is terminal, an explicit approval or HUMAN-HOLD requires user input, a proven blocker leaves no ready work, or a merge decision requires the user. Hooks are advisory and never replace this watch cycle.
