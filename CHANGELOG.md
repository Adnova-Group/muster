- **`tallyReview` expresses worker exhaustion as a blocking status (backlog item `tally-worker-exhaustion-contract`).** Live incident (2026-07-19 Codex dogfood): an agent-watch budget killed a required reviewer mid-run, and the orchestrator -- with no vocabulary in the tally contract to say "this reviewer never verdicted" -- fed synthetic FAIL/blocker-shaped strings into `muster_tally`; `tallyReview` (`src/review.js`) only ever parses the `findings[].severity` vocabulary (`blocker`/`risk`/`nit`), so those synthetic strings were silently skipped exactly like any other unrecognized severity, and the gate wrongly answered `blocked: false`, forcing the orchestrator to improvise its own halt outside the deterministic gate math. A reviewer entry may now carry a top-level `status: "exhausted"` (killed/ran out of budget) or `"absent"` (never dispatched/never responded), naming the WORKER's own failure to deliver a verdict -- orthogonal to and never parsed alongside `findings`. Such an entry always forces `blocked: true` with a named reason in a new `blockedReasons` array (e.g. `"reviewer b exhausted before verdict"`) and is recorded in a new `exhausted` array; it is never a silent skip and never counted toward `counts`/`blockers` as if it cast a real PASS or FAIL vote -- exhaustion always overrides silence, even when the entry carries a spoofed `severity: "blocker"` finding. An unrecognized `status` value is ignored (same "unknown vocabulary is skipped, never guessed" discipline the gate already applies to an unrecognized severity), and entries without `status` are fully backward compatible. The gate has no majority/quorum computation to begin with, so an exhausted/absent entry trivially cannot count toward any quorum -- documented in `src/review.js` for if one is ever added. Red-first TDD in `test/review.test.js` reproduces the exact dogfood fixture (an exhausted reviewer carrying only synthetic FAIL noise instead of a verdict) plus the spoofed-blocker, absent-status, mixed-reviewer, unrecognized-status, and backward-compatibility cases. `cowork/mcp-server.mjs`'s `muster_tally` tool description now documents the contract for MCP callers (`test/claude-parity.test.js`'s Claude-surface pin re-derived to match), and `website/reference/commands.md`'s `tally`/`pick` row is expanded to describe both the any-blocker gate and the new exhaustion contract.

### Changed
- **Codex agent-watch invariant is liveness-aware, with per-class extension ceilings (backlog item `codex-agent-watch-review-budget`).** The flat "three consecutive silent heartbeats = kill" rule conflated "no receipt" with "hung": a live incident (2026-07-19 dogfood) interrupted a healthy `gpt-5.6-sol/high` `muster-reviewer` mid-review before its PR -- review-class reasoning routinely exceeds 3 silent minutes with zero mailbox receipts (sol/high averages ~10min/task per DeepSWE, and its rollout showed a live, 5/5-conformant thread), which halted an otherwise-green run and doubled wall-clock. `codex/skill-adapter.md`'s "## Agent watch invariant" section now checks the already-reconciled `collaboration.list_agents` state at the third silent heartbeat before interrupting: a thread actively in a turn (running) is extended instead of killed, bounded by a hard per-class ceiling -- 10 consecutive silent heartbeats for review/strategy-class workers (`muster-reviewer`, `wsh-code-reviewer`, `muster-strategist`, `wsh-security-auditor`), 6 for mechanical/implementation lanes. A worker whose thread is NOT in a turn (idle, completed, or failed) at three silent heartbeats, or one that outlives its ceiling, is interrupted/recorded/escalated exactly as before -- genuinely dead workers still die fast. `scripts/build-codex.mjs`'s `agentWatchProtocol` const carries the identical prose into the orchestrator and every mode/alias `SKILL.md` (no single source of truth between the two copies, so both are hand-kept in sync -- a comment now says so). `scripts/check-codex.mjs`'s and `test/codex-workflows.test.js`'s watch-marker pins are extended to cover the liveness check and both ceilings, red-derived against the old flat-kill text before the prose change landed. `docs/research/codex-cli.md`'s `[CODE-VERIFIED]` description of the invariant is reworded to match.

- **Spec gate allows a second amendment for disjoint round-2 findings (backlog item `spec-gate-amendment-policy`).** The one-amendment ceiling previously hard-escalated on ANY round-2 spec-gate FAIL, unable to distinguish "the spec is still broken the same way" from "the spec improved and the gate dug deeper" -- a live incident (2026-07-18 Codex dogfood, `$muster-go` run) had round 1 FAIL on test-ownership + underspecified contracts, an amendment fixed those, then round 2 FAIL on a wholly distinct, newly-surfaced finding (no executable caller consumes the receipts), aborting an otherwise-salvageable run. `plugin/commands/go.md` step 4 now itemizes findings per round in the spec-gate return contract and compares round 2's list against round 1's before deciding: any round-2 finding that restates or is a subset of an unresolved round-1 finding still hard-aborts (repeated defect), but a round-2 FAIL whose findings are ALL disjoint from round 1's (no repeats) earns a second amendment instead of escalating immediately. A round-3 FAIL always hard-aborts unconditionally, regardless of disjointness -- rounds are capped at 3 total (the initial dispatch plus at most two amendments). The repeated-vs-disjoint determination is recorded in STATE per finding (glass box), so the amend-vs-escalate decision is auditable, not a black box. Step 7's escalation trigger list and the Routine-mode escalation bullet are reworded to match ("a spec-gate hard abort" instead of "a second FAIL"), and `docs/architecture.md`/`website/reference/architecture.md`'s Go paragraphs are synced. The ceiling was prose-only (`plugin/commands/go.md`) -- no `src/`/`eval/` code encoded a round cap (`src/gate-cadence.js`'s `specGateRounds` only tracks whether the gate runs at all, 0 or 1, unrelated to this amendment loop) -- so this is a doc/prose change with no code paths to test. `test/claude-parity.test.js`'s Claude-surface pin is re-derived to match the edited `plugin/commands/go.md`.

### Removed (breaking)
- **Enforcement-model redesign: the wave-guard, the per-turn scale-gate, and the todo-driving gate are gone.** All three were repeatedly patched for field-reported false positives on this very branch (most recently a session-engagement marker bolted onto the scale gate) rather than fixed at the root, because the root problem was the model itself: a hard main-loop deny gated on file-system markers cannot distinguish "orchestration-scale drift" from "legitimate concurrent work in a repo/session muster never touched," and repeated-injection kill-switch guidance (`MUSTER_WAVE_GUARD=off`, `MUSTER_SCALE_GATE=off`) trained agents to disable the enforcement rather than trust it. Rather than patch a fourth time, the enforcement model itself changed: **enforcement now follows the run's EXTERNAL effects, not the orchestrator's own in-repo edits.** Removed entirely:
  - **Wave-guard.** Denied any main-loop `Edit`/`Write`/`NotebookEdit` or high-confidence Bash file write while `.muster/wave-active` existed (`deny()`/`denyBash()`/`warnAllow()`, the `EDIT_TOOLS` deny branch, the Bash write-target deny branch, the `STALE_MS` orphan-detection logic, all in `pre-tool-use.js`). `.muster/wave-active` may still be written by skills; this hook no longer reads it.
  - **Per-turn + post-run scale gate's deny path.** Denied the Nth distinct inline file edit in a turn once a session was deemed "engaged" (`applyScaleGate`'s deny/threshold paths, `denyScale`/`warnScaleAllow`, the per-turn `budgetFile` machinery). The underlying cumulative cross-turn counter survives as a **warn-only border invitation** (see Added, below) — it no longer needs an engagement heuristic because it never denies.
  - **The session-engagement marker** (`inline-budget.js`: `engagedFile`, its `SessionStart` clearing, its tests) — added earlier on this branch specifically to scope the scale-gate's deny to "sessions that ran muster," now dead along with the deny path it scoped.
  - **Todo-driving gate** (`plugin/hooks/todo-gate.js`, its `hooks.json` `Task|Agent` matcher, its tests) — denied a subagent-wave dispatch without a native `TodoWrite`/`TaskCreate`/`TaskUpdate` since run start. Removed with a veto flag noted: it was gameable with a throwaway todo (enforced visibility, not compliance) and added dispatch-time transcript-scan latency for that guarantee. Todo-driving is now `plugin/skills/orchestrator/SKILL.md` discipline, unenforced.
  - **Envs `MUSTER_WAVE_GUARD`, `MUSTER_SCALE_GATE`, `MUSTER_TODO_GATE`, `MUSTER_NUDGE_EVERY`, `MUSTER_PRINCIPLES_EVERY`** — removed everywhere (code, hook messages, README, docs/architecture.md, website reference/install docs). `MUSTER_INLINE_SCALE` survives, now solely as the border-invitation's file-count threshold.
  - **The periodic every-N-turns nudge tier** (`user-prompt-submit.js`: the short nudge at N turns, the full principles payload at N*K turns) — habituated, and injected on turns that had nothing to do with routing.
  - **`SessionStart`'s full-payload guidance** (principles + all seven verbs + routing policy + project sniff) — trimmed to a one-line pointer (`muster available; /muster:plan for orchestration-scale work`); marker-cleanup duties (stale `run-active`/`wave-active`, drift-counter, directive-marker resets) are unchanged.

## [0.2.5] - 2026-06-10

### Added
- **Doctor `install-integrity` check** — verifies each registered muster entry's `installPath` actually exists and contains `hooks/hooks.json`. Catches the field failure where `installed_plugins.json` records a successful install but the plugin-cache copy silently never happened, leaving the plugin inert (no hooks → no routing) while the staleness check reports healthy. Failure detail names the missing path and the uninstall/reinstall remediation.

## [0.2.4] - 2026-06-09

### Added
- **`MUSTER_MAX_TIER` env cap** — `capTier(tier, cap)` (exported from `src/model.js`) enforces a model-tier ceiling: if the resolved tier sits above the cap in `MODEL_TIER_ORDER`, the cap is returned instead. `modelForRole` applies the cap as its final step so it flows through capabilities → crew manifests → dispatch. Invalid or unset cap is a no-op (fail-open). Example: `MUSTER_MAX_TIER=opus` disables Fable and routes peak-judgment roles to Opus; `MUSTER_MAX_TIER=sonnet` for budget mode.
- **Bash wave-guard (extends PreToolUse hook)** — the PreToolUse hook (first shipped in 0.2.3 for `Edit|Write|NotebookEdit`) now also matches `Bash` commands. When a wave is active, Bash commands are inspected by `bashWriteTarget()` (pure function in `plugin/hooks/bash-write-target.js`, unit-tested without spawning) and denied only on high-confidence file writes: `sed -i`, `tee` to a non-exempt target, and `>` / `>>` redirects to non-exempt targets. Everything else allows (fail-open). Exempt targets: `/dev/*`, `/tmp/*`, `.muster/*`. Deny reason names the matched pattern and notes `MUSTER_WAVE_GUARD=warn` as the false-positive escape hatch. `MUSTER_WAVE_GUARD` values: `deny` (default), `warn` (allow with reminder), `off` (disable guard). Known limitations: unbalanced quotes and heredoc bodies are not fully parsed (balanced quoted strings are handled).
- **SHA-pinned vendor sources** — github sources in `vendor/manifest.yaml` are pinned to full commit SHAs (`wshobson/agents`, `open-gsd/gsd-core`); `muster vendor` fetches a pinned SHA via init/fetch/checkout (`cloneCommandsFor`). Updating an upstream is now a deliberate act: bump the SHA, re-vendor, and review the full regenerated diff before committing (see CONTRIBUTING "Vendored content").
- **Fable tier** — `modelForRole` routes peak judgment (tournament judge, architecture-review) to `fable`, degrading to `opus` when unavailable (`fallbackModelFor`); ordered tier policy exported as `MODEL_TIER_ORDER`/`maxTier`.
- **Crew model binding** — every non-inline crew member carries its `model` in the manifest (`makeStage`), and `manifest validate` rejects members without a known tier; the orchestrator passes the bound model as the Agent dispatch override.
- **`muster steer` subcommand** — deterministic steering classification over the CLI; orchestrator prose no longer references un-shipped `src/` functions.
- **Doctor checks** — hooks.json integrity (events + script paths) and installed-plugin staleness vs repo version, with remediation steps.
- **Generated-artifact drift test** — vendored agent frontmatter models must match `modelForRoles(roles)`; caught and fixed 3 stale `wsh-*` agents.

### Fixed
- All `npx muster` invocations in plugin prose now use `npx -y @adnova-group/muster` (the bare name resolves an unrelated registry package).
- `install` run from the ephemeral npx cache now recommends the GitHub marketplace (`Adnova-Group/muster`) instead of a prunable local path.
- Review-gate iteration cap (3) enforced in code (`reviewGateState`); vendor version-dir selection picks max semver (hash dirs no longer win); `skills` detection populated in the harness scan; `fileURLToPath` for repo-root resolution.
- Vendor pipeline hardening: id/repo/ref format validation, clone-root and repoRoot containment, tools allowlist for vendored agents.
- Stale "opus = heavy judgment" prose corrected to fable across README, docs, website, and command files; duplicate/ambiguous skill descriptions de-duplicated (gsd-verify-work, interview, domain-router); sp-debug no longer claims the implement role.

## [0.2.3] - 2026-06-09

### Added

- **In-session drift reinforcement (`UserPromptSubmit` hook).** A new plugin-native hook
  re-asserts muster mode on a turn cadence so a session stops reverting to default inline Claude
  behavior. Short nudge every `N` turns (`MUSTER_NUDGE_EVERY`, default 3); full
  principles + verbs every `N*K` turns (K = `MUSTER_PRINCIPLES_EVERY`, default 3 -- at defaults:
  nudge every 3, full every 9). Per-session turn counter in `os.tmpdir()`; fully self-contained and
  fail-safe (always valid JSON, exit 0). Compaction still re-fires `SessionStart` as a backstop.
- **Default-routing posture (`ROUTING_POLICY`).** Injected guidance now states that, in a
  muster repo, actionable prompts should be driven through the verbs (and copy/content through the
  humanizer) by default, while conversational turns fall through — no need to prefix every task
  with `/muster`. Carried in the `SessionStart` payload, the periodic full payload, and a condensed
  clause in the short nudge.
- **Orchestrator iron rule + `PreToolUse` wave-guard hook.** `plugin/skills/orchestrator`
  leads with the iron rule: each wave task MUST be dispatched via the Agent tool before any edit,
  with an announce-to-STATE requirement so inline work is auditable. The `PreToolUse` hook
  (`plugin/hooks/pre-tool-use.js`) now ships as the hard harness-level gate. Decision order:
  (1) subagent calls always allowed; (2) `.muster/` STATE writes always allowed; (3) allow if no
  wave marker; (4) allow if marker is stale (>60 min); (5) honour `MUSTER_WAVE_GUARD`:
  `off` = silent allow, `warn` = allow with reminder, unset or `deny` = deny.

### Changed

- **Shared `plugin/hooks/guidance.js` is the single source of truth.** `session-start.js` was
  refactored to consume `PRINCIPLES`/`VERBS`/`ROUTING_POLICY`/`detect` from it instead of inlining
  the text, so the two hooks can never drift apart.

## [0.2.0] - 2026-06-08

### Added

- **`muster prioritize`: three new scoring models** (#2). Beyond the existing
  RICE model, the deterministic scorer now offers `--model ice`
  (impact times confidence times ease), `--model wsjf` (cost-of-delay divided by
  job-size), and `--model weighted` (Aha-style weighted scorecard: the sum of
  weight times score across custom criteria). Each fails loud on non-finite,
  non-positive, or zero-denominator inputs, matching RICE's discipline. The four
  models share one `scoreAndRank` scaffold, so they all produce the same shape:
  2-decimal scores, score-descending order with an ascending-name tie-break, and
  1-based ranks.
- **Roadmap pipeline: optional GitHub Projects board output** (#4). The
  `roadmap-prioritization` skill can now push the prioritized initiatives onto a
  GitHub Project board with status columns by tier (Now / Next / Later), in
  addition to the doc and issues. It degrades gracefully, skipping with a note
  when `gh project` access is unavailable, and reuses an existing board before
  creating one.

### Changed

- The `prioritize` dispatcher's unsupported-model error and the CLI usage strings
  now list every supported model (`rice`, `ice`, `wsjf`, `weighted`).
- `docs/architecture.md` documents all four prioritization models.

### Documented

- **Orchestrator generic-subagent fallback** (#1). The orchestrator skill now
  explicitly documents the degraded path: when a role resolves to an agent
  provider whose type is not yet dispatchable in the running session (for example,
  plugin agents installed before a restart), it falls back to a generic subagent
  with the resolved provider's brief injected and the role's model override still
  applied.

## [0.1.0] - 2026-06-08

### Added

- Initial public release. A glass-box, multi-domain agentic orchestrator for
  Claude Code: it detects the project, discovers installed capabilities, resolves
  each role to the best available provider, and runs a crew toward an outcome with
  every decision inspectable. Ships the `muster` CLI (deterministic, no LLM calls),
  the plugin (agents and skills), the capability and domain catalogs, the domain
  pipelines, and the glass-box output style. Runs on bare Claude Code and improves
  as more tools are installed.
