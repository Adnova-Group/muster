# Contributing to Muster

Thanks for your interest in Muster. This guide covers how to set up the repo, how it is laid out, and how to add the kinds of things contributors most often add.

## Development setup

Clone the repo, then install dependencies:

```bash
git clone https://github.com/OWNER/muster.git
cd muster
npm install
```

There is a single runtime dependency (`yaml`), so install is quick.

Run the test suite with:

```bash
npm test
```

That runs the [`node:test`](https://nodejs.org/api/test.html) suites under `test/`. Node 20 or newer is required (see `engines` in `package.json`).

There is no build step. The CLI runs the `src/` tree directly:

```bash
npx muster <verb>
# or, equivalently
node src/cli.js <verb>
```

Try `node src/cli.js detect` or `node src/cli.js capabilities` to see it work against the current directory.

## Project layout

| Path | What lives here |
| --- | --- |
| `src/` | The deterministic CLI. Plain Node modules, no LLM calls. Each verb maps to a module dispatched from `src/cli.js`. |
| `plugin/` | The model-facing surface: `commands/`, `skills/`, and `agents/`. This is what Claude Code loads. |
| `catalog/*.yaml` | The capability catalog. Maps provider IDs to roles and ranks so the router can pick a crew. |
| `pipelines/*.yaml` | Domain pipelines (one file per pipeline). Each defines the phases for an outcome like a PRD or a roadmap. |
| `vendor/manifest.yaml` | The vendoring source of truth. Lists upstream skills and agents to pull in and the role each maps to. |
| `output-styles/` | Output style definitions for the plugin. |
| `test/` | The `node:test` suites. Unit tests sit alongside `integration.*` and `smoke` suites. |

The split between `src/` and `plugin/` is the important one. `src/` is deterministic code that you can run and test without a model in the loop. `plugin/` is the prompt-and-skill layer the model reads. When something can be computed (routing, scoring, wave ordering, validation), it belongs in `src/` as code, not in a prompt.

## How to add things

### A new pipeline

Add `pipelines/<id>.yaml`. The shape, drawn from an existing pipeline:

```yaml
id: prd
domain: pm
title: Product Requirements Document
match: [prd, product requirements, product spec]
phases:
  - { id: intake, role: brainstorm, desc: "clarify problem, audience, success metrics" }
  - { id: draft,  role: author,     desc: "write the document" }
  - { id: review, role: code-review, desc: "adversarial review gate" }
gate:
  criteria: [problem-clarity, evidence, scope-discipline]
  floor: 2
  pass_total: 16
```

Required keys: `id`, `domain`, `title`, `match` (phrases the router matches against an outcome), `phases` (each with `id`, `role`, `desc`), and a `gate` (review criteria, a per-criterion `floor`, and a `pass_total`). Add a test under `test/` that loads the pipeline and asserts it routes and validates. The content pipeline suites are a good template.

### A new agent

Two routes:

1. **Clean-room muster agent.** Write the agent definition in `plugin/agents/<id>.md`, then add a `kind: agent` entry to `catalog/agents.muster.yaml` with its `id`, `roles`, `rank`, and a `description`. The muster-authored agents (`muster-builder`, `muster-reviewer`, and so on) show the pattern.
2. **Vendor an upstream agent.** Add an item to the relevant source in `vendor/manifest.yaml` with `as: agent` and the `roles` it fills, then re-vendor (see below). This writes both `plugin/agents/<id>.md` and an entry in `catalog/agents.generated.yaml`.

Pick the muster-authored route for behavior specific to muster. Pick vendoring when a good upstream agent already exists and you only need to map it to a role.

### A new skill

Add `plugin/skills/<name>/SKILL.md`. Each skill is its own directory with a `SKILL.md`. The existing skills (`router`, `interview`, `tournament`, and others) are the reference. If the skill should be selectable by the capability catalog, give it a catalog entry (`catalog/builtins.muster.yaml` for muster-authored built-ins, or vendor it via the manifest).

### A new CLI verb

Three steps:

1. Add the logic as a module in `src/` (a pure function is easiest to test).
2. Wire a dispatch branch into `main()` in `src/cli.js`, following the existing `if (cmd === "...")` pattern. Update the usage string at the bottom too.
3. Add a test under `test/`. Test the `src/` module directly where possible; CLI args parsing helpers live in `src/cli-args.js` so tests can import them without triggering dispatch.

## Vendoring

The built-in skills and agents are generated from `vendor/manifest.yaml`. Regenerate them with:

```bash
node src/cli.js vendor
```

Generated files carry a `GENERATED by 'muster vendor' ... do not edit` banner (for example `catalog/agents.generated.yaml`, `catalog/builtins.generated.yaml`, the vendored `plugin/` files, and `NOTICE`). Do not edit those by hand. Edit `vendor/manifest.yaml` and re-vendor instead, otherwise your change is lost the next time someone vendors. The manifest validates before it writes, so a malformed entry fails loudly rather than producing a broken catalog.

## Testing and style

- **Test first.** Write a failing test that captures the intent, then make it pass. A test that only mirrors the current behavior is not worth much, so encode the reason the behavior should hold.
- **Fail loud.** Validation paths reject bad input with a clear message and a non-zero exit. Do not swallow errors or return a silent default where a caller would want to know something was wrong.
- **Keep the CLI deterministic.** `src/` does not call a model. If a transform can be expressed in code, write the code. The model-facing reasoning lives in `plugin/`.
- **Match the house style.** Terse, direct prose in docs and prompts. No marketing language. Look at neighboring files before adding a new one.

## Pull request conventions

- Keep PRs small and focused on one change.
- Write conventional-commit-style messages (`feat:`, `fix:`, `docs:`, `refactor:`, and so on).
- Make sure `npm test` is green before opening the PR.
- Do not add a new dependency without discussing it first. The lean dependency tree is a feature.
