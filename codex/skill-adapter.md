# Codex harness binding

Apply these bindings before following any ported Muster workflow. They override legacy harness names in the source instructions while preserving the workflow's domain logic, gates, approvals, and return contracts.

- Treat `Agent` and `Task` calls as Codex subagent dispatch. For a resolved `kind: agent` provider, call `collaboration.spawn_agent` with the ordinary `task_name`, `message`, `fork_turns: "none"`, and `agent_type: "<exact chosen.id>"`. A positive context fork is allowed only when the user explicitly requests one; never use `"all"`. Codex rejects a named `agent_type` combined with a full-history fork because full-history agents inherit the parent's type/model/effort. `agent_type` is a Codex runtime extension and may be absent from a simplified displayed tool signature; include it anyway. The installed custom-agent TOML pins the role's model, reasoning effort, sandbox, and instructions. Only an actual rejected `spawn_agent` tool call proves the type unavailable; inspecting a displayed signature or assuming an omitted field is unsupported is not an attempt. If the call rejects that type, fail closed with a registration diagnostic rather than silently losing model policy through a general agent. For any resolved `kind: skill` provider, run `node ${PLUGIN_ROOT}/runtime/resolve-skill-provider.mjs <chosen.source> <chosen.id>` so provenance and the safe kebab-case id are centrally validated before a path or invocation is constructed. When `source === "builtin"`, verified stdout is the bundled workflow to inject into a general subagent's brief; load every relative bundled asset through the same command's optional third asset argument. When `source === "installed"`, verified stdout is the supported explicit `$skill-id` invocation contract; invoke that already-enabled Codex skill and never substitute Muster's bundled fallback. Use a general subagent directly for MCP/inline providers. Preserve dependency waves, isolation, return contracts, and the configured Codex thread concurrency; dispatch only manifest-ready, nonredundant workers. Workers are leaves and must not spawn descendants unless an approved manifest explicitly delegates nested orchestration. Send a minimal dispatch packet containing only the task, relevant success criteria, owned/frozen paths, worktree paths, dependency receipts, required provider/skill brief, and return contract; never attach unrelated plan items, capability inventories, or prior transcripts. Every brief must set a 25-step ceiling, allow at most one follow-up turn, and prohibit broad test suites until final verification. Interrupt and record a budget escalation when a worker exceeds that ceiling or remains live after three 60-second heartbeats without a receipt.
- Codex subagent dispatch has no cwd field. Every brief for worktree-scoped work must therefore name the absolute `WORKTREE CWD`, the absolute manifest/STATE paths inside that worktree, and require the child to use that directory as the `workdir` for its first verification command and every later tool call. The child must not read or mutate the parent checkout's `.muster` artifacts. Treat a parent-checkout/worktree manifest mismatch as a dispatch defect, not as authority to choose one silently.
- Treat `AskUserQuestion` as Codex's structured user-input mechanism when available. Otherwise ask one concise plain-text question and wait; never infer approval.
- Treat `TodoWrite` as Codex's task-board interface. For the approve-first plan surface specifically, do not map `ExitPlanMode` to `TodoWrite`: follow the ported command file's own Codex-native branch instead (`permission_mode: "plan"` plus the bundled system `plan` skill — docs/research/codex-cli.md §1, §4.2, §5.2; the same per-harness table lives as tested code in `src/plan-surface.js`).
- Treat `Skill` calls as explicit `$skill-name` invocations.
- Map `/muster:run` to `$muster-plan`, `/muster:autopilot` to `$muster-go`, `/muster:plan-backlog` to `$muster-plan-backlog`, `/muster:sprint` to `$muster-go-backlog`, and the diagnose, audit, runner, and capture commands to their matching `$muster-*` skills.
- `$ARGUMENTS` means the user's remaining prompt. `${PLUGIN_ROOT}` and `${CLAUDE_PLUGIN_ROOT}` both mean the installed Codex plugin root.
- Run every legacy `npx -y @adnova-group/muster ...` instruction through the self-contained bundled CLI instead: `node ${PLUGIN_ROOT}/runtime/muster.mjs ...`. Never contact the npm registry for a plugin workflow. Add `--codex` to `capabilities`, `match`, and `manifest validate` so provider and skill ids are resolved against the enabled Codex inventory.
- When a legacy workflow runs `muster capabilities`, use the bundled CLI with `capabilities --codex` so routing is based on enabled Codex plugins, skills, MCP servers, and profiles rather than Claude state.
- Translate `Read`, `Write`, `Edit`, `Bash`, `Grep`, and `Glob` to the available Codex tools while respecting the generated agent profile's read/write policy.
- When Claude Code itself is the actual subject of a workflow, retain that scope rather than claiming a nonexistent Codex equivalent.
- `muster install codex` merges Muster's lifecycle hooks into the selected project or user `hooks.json` layer. After the user trusts their exact definitions in `/hooks`, they provide context, diagnostics, and supported policy warnings. Codex DOES execute plugin-bundled hooks (landed 0.128, default-on since 0.131), so Muster's Codex plugin is deliberately hooks-free — its hooks come only from the `hooks.json` layer; a plugin that shipped a `hooks/hooks.json` would double-fire on top of it (the hook-bombardment regression). Treat todo and spawn policy as advisory; every write-capable wave must still run in an isolated git worktree.

## Agent watch invariant

<!-- prompt-lint-disable GUARD-IDK-001: Explicit terminal conditions prevent abandoned live agents while preserving approval, HUMAN-HOLD, blocker, and merge-decision stops. -->

After every dispatch, retain every canonical agent id returned by `collaboration.spawn_agent` and immediately call `collaboration.wait_agent` with a timeout of at most 60 seconds. A message or completion receipt wakes the watch immediately. After each wake, process mailbox receipts first, call `collaboration.list_agents` exactly once to reconcile live state, and dispatch any newly ready work. Never tight-poll.

Three consecutive heartbeats without a receipt are a liveness checkpoint, not an automatic kill: before interrupting, read that heartbeat's own `collaboration.list_agents` reconciliation for the worker's thread state. A thread actively in a turn (running) is THINKING, not hung -- extend the watch instead of killing it [DeepSWE sol/high averages ~10min/task; the 2026-07-19 dogfood incident interrupted a healthy gpt-5.6-sol/high muster-reviewer mid-review on a receipt-free but live thread]. This check is not a one-time gate at heartbeat 3 -- re-run the same `list_agents` reconciliation at EVERY silent heartbeat between 3 and the worker's ceiling, catching a thread that goes idle or fails partway through an extension at the next heartbeat rather than only at the ceiling. A thread NOT in a turn (idle, completed, or failed) at any checked heartbeat is exhausted exactly as before: interrupt the worker, record the incomplete task in STATE, record `{reviewer: <name>, status: "exhausted"}` in the tally input for the interrupted worker, and escalate or continue locally only when safe.

Extension is bounded by worker class. Review/strategy-class workers (`muster-reviewer`, `wsh-code-reviewer`, `muster-strategist`) get a hard ceiling of 10 consecutive silent heartbeats, sized to sol/high's measured task time. `wsh-security-auditor` is pinned to sol/XHIGH (DeepSWE mean ~13.3min/task -- slower than the sol/high figure the 10-heartbeat ceiling above was sized to) and gets its own hard ceiling of 14 consecutive silent heartbeats instead. Mechanical/implementation workers keep the 3-heartbeat exhaustion above once `collaboration.list_agents` shows no active turn, and get the same in-turn extension bounded by a hard ceiling of 6 consecutive silent heartbeats. On ceiling exhaustion the existing protocol applies unchanged: interrupt the worker, record the incomplete task in STATE, record `{reviewer: <name>, status: "exhausted"}` in the tally input for the interrupted worker, and escalate or continue locally only when safe.

Respect the configured `agents.max_threads`; Muster must neither lower nor raise it. Spawn with `fork_turns: "none"` unless the user explicitly requests a context fork. Every brief sets a 25-step ceiling, permits at most one follow-up, and defers broad suites to final verification. Do not send the final answer or clear state while executable work remains, but worker budget exhaustion is a terminal escalation condition rather than permission to wait forever. Hooks are advisory and never replace this watch cycle.
