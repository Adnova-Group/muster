You are muster's runner: one unattended work cycle — resume or claim exactly ONE item, run it, leave a receipt, stop.

Respond with the cycle's receipt (idle / resumed / claimed, disposition executed, gate summary, escalation if any) written to STATE.

<source>$ARGUMENTS</source>

**Run-active lifecycle:** Write `.muster/run-active` at invocation start (before step 1) -- the mode/run-in-progress marker the `PreToolUse` hook uses to scope the scale-gate. Remove it after step 6. `SessionStart` on a fresh session clears a stale marker automatically. `.muster/run-active` is owned by THIS outer mode (runner), not by the per-item autopilot pass: step 4's "run autopilot steps 1-8" excludes autopilot's own run-active preamble — the marker is written once at this cycle's invocation start and removed only at this cycle's own exit (step 6), never mid-item.

Load the **coordination** skill before step 2 — every cycle shares state with other runners and humans.
Its HYGIENE PREFLIGHT step runs `node src/cli.js hygiene --reap` once per cycle, before any claim — a
dead run (crashed mid-cycle) must not strand a zombie provider process or a stuck coordination claim for
the next cycle to trip over.

1. **Resolve the source** — `$ARGUMENTS` empty or a path defaults to `.muster/backlog.md` (Binding B); `issues:<label>` uses Binding A; `linear:<team key or project>` uses Binding C. A missing file, a `gh` failure, or a Linear MCP failure is reported and the cycle stops — nothing claimed.
2. **Resume check** — scan for a BLOCKED or HUMAN-HOLD answer BEFORE any new claim, a single unordered pass (per the coordination protocol): a BLOCKED item resumes on ANY reply — Binding B: a STATE `ANSWER <slug>: ...` line newer than the matching `BLOCKED ... <slug>` receipt; Binding A: a non-`MUSTER `-prefixed (human) reply newer than the `MUSTER BLOCKED` comment. A HUMAN-HOLD item's resume depends on the binding: Binding A resumes on a reply whose comment author's `.user.login` matches the `authorizer=<login>` recorded in the `MUSTER HUMAN-HOLD` comment (any other reply is inert and leaves it held) — GitHub authenticates that authorship, so this unattended cycle can trust it. Binding B has no such authentication (a STATE line is just text any writer can produce), so this UNATTENDED cycle never resumes a Binding B `{human-hold:}` item — it stays **permanently parked**; only an attended `/muster:sprint` session (its own AskUserQuestion-gated resume, per the coordination protocol) can clear it. A resumed item (Binding A human-hold, or either binding's plain BLOCKED) resumes (claim receipt noted) in place of step 3, then goes straight to step 4.
3. **Claim** — per the coordination protocol: skip already-claimed and escalated items, and respect the 2-failure retry cap (redirect to blocked rather than reclaiming). Claim the FIRST available item only, never more. An item whose requirements can't be understood from its text is BLOCKED with a question receipt instead of claimed — if you're unsure what it's asking, say so in the receipt rather than guessing. Nothing available (empty, or every item claimed/escalated/blocked) → write an `idle` receipt to STATE and STOP.
4. **Execute** — run the resumed/claimed item through autopilot steps 1-8 (branch, detect, route, spec gate, plan, orchestrate waves, escalation check, finish), applying BOTH of sprint.md's per-item overrides: step 3's info-gap resolves with best-effort defaults instead of the interview (gap signals recorded); step 8's merge decision is FORCED to `pr` regardless of the declared value (`merge-local`/`merge-push`/`ask` all coerce to it) — this cycle never touches the base branch unattended, and the coercion (declared value → `pr`) is noted in the receipt.
5. **Receipts + ledger** — leave the item's terminal receipt (DONE/BLOCKED/HUMAN-HOLD/FAILED) per the coordination protocol, then edit this runner's ONE ledger heartbeat in place (last-seen, last item, result). **On escalation** (step 4's escalation check trips): post a `FAILED` receipt (attempt-counted, the coordination protocol's existing format) AND mark the item escalated so step 3's "skip escalated" has a concrete artifact to check — Binding B: append `{escalated: <runId or date>}` (sprint.md's exact convention, reused not reinvented); Binding A: move the issue to `agent:needs-input` with a question comment (reusing the BLOCKED convention — there is no separate escalated label).
6. **Stop** — exactly one item per cycle, however it resolved (done/escalated/blocked/idle). The schedule provides the loop, not this mode.

**Scheduling** — fire `/muster:runner` from a Claude Code Routine on a timer, or from cron invoking the Claude Code CLI (e.g. `claude -p "/muster:runner issues:agent:todo"`): the verified-safe standing-cadence default. Match the backlog's arrival rate, not faster — an idle cycle is cheap, but firing far ahead of new items just piles up idle receipts; start around 15-30 minutes and widen if idle cycles dominate the ledger, tighten if items queue up unclaimed. Native `/loop /muster:runner <source>` is **not** a standing-loop mechanism for this command: as of Claude Code v2.1.196 a scheduled/`/loop` fire does not execute a `disable-model-invocation: true` command — it reaches Claude "as plain text instead of executing" — so the second and later `/loop` fires re-run nothing here (confirmed 2026-07-20 against `scheduled-tasks.md`'s "allowed to invoke on its own" list and `skills.md`'s `disable-model-invocation` row; both cited in docs/research/claude-code-cli.md §1). `disable-model-invocation: true` stays on this command by decision — the routing-safety rationale that set it (keep the model from auto-invoking a heavy unattended runner) stands; it is not flipped to buy back `/loop`. For a native self-continuing loop, use Claude Code's `/goal`: it sets a completion CONDITION re-checked each turn until a model confirms it (session-scoped), distinct from `/loop`'s time-interval re-fire. Safety inventory (Routine or cron): pr-only (step 4 never merges/pushes to base), the 2-failure retry cap bounds reclaim loops, and the claim lock (Binding A's comment-race window / Binding B's claim-then-verify) makes concurrent firing safe alongside `/muster:sprint`, other scheduled runners, and humans on the same backlog.

Glass box: the resume/claim decision, the executed disposition (and any pr-coercion), and the receipt trail are all recorded in STATE as the cycle runs.
